## Finding 1 — High: Unchecked arithmetic in `transfer` allows arbitrary balance minting and full vault drain

**Tool Severity:** High Risk
**Triage Decision:** Valid — core vulnerability correctly detected.

`transfer` wraps both balance updates in an `unchecked` block and never
verifies that `balances[msg.sender] >= amount` before subtracting:

```solidity
function transfer(address to, uint256 amount) external {
    unchecked {
        balances[msg.sender] -= amount;
        balances[to] += amount;
    }
}
```

An attacker with a zero balance can call `transfer(anyAddress, 1)`, which
causes `balances[attacker]` to underflow to `2^256 - 1`. Since `withdraw`
trusts the mapping without additional validation, that fabricated balance
is accepted immediately:

```solidity
function withdraw(uint256 amount) external {
    require(balances[msg.sender] >= amount, "Insufficient balance");
    ...
}
```

Once honest users have deposited any ETH, the attacker calls `withdraw`
against the forged balance and drains the entire contract. The recipient-side
addition (`balances[to] += amount`) is also unchecked, allowing an attacker
to arbitrarily overwrite any account's balance — including zeroing out
legitimate users.

This is the primary vulnerability the contract was designed to demonstrate.
High severity is correct — one transaction from a zero-balance address is
enough to set up a complete drain with no preconditions beyond a nonzero
contract balance.

AuditAgent's analysis is precise: it identifies both the underflow on the
sender side and the unconstrained overflow on the recipient side as separate
but compounding consequences of the same root cause.

Note: `withdraw` also uses `unchecked` for its own subtraction, but it is
protected by the preceding `require` check, so that path is safe. AuditAgent
correctly did not flag it.

## Finding 2 — Low: Non-specific Solidity pragma version

**Tool Severity:** Low Risk
**Triage Decision:** Valid — correctly classified.

`pragma solidity ^0.8.0;` allows compilation against any `0.8.x` compiler.
In practice this means the generated bytecode can differ between environments
depending on which compiler version is resolved, potentially introducing
subtle changes in optimization behavior or emitted opcodes.

The fix is straightforward: pin to a specific version (e.g.,
`pragma solidity 0.8.20;`). The risk is low in isolation — no attacker can
exploit a floating pragma — but it introduces non-determinism in production
deployments. Correct as Low.

## Finding 3 — Low: PUSH0 opcode compatibility issue

**Tool Severity:** Low Risk
**Triage Decision:** Valid in principle — not exploitable in this lab context.

Solidity `^0.8.0` can resolve to `>=0.8.20`, which emits the `PUSH0` opcode
introduced in the Shanghai EVM upgrade. Chains or testnets that have not
adopted Shanghai will reject the bytecode at deployment. This is a real
portability concern for production contracts targeting multiple chains (e.g.,
Polygon, Arbitrum, older Ethereum forks).

For this lab running on a local Foundry/Anvil environment the issue is
irrelevant, but the finding is legitimate in a production context. Low
severity is appropriate — it does not represent a logic vulnerability.

## Finding 4 — Best Practices: Missing events for all state-changing operations

**Tool Severity:** Best Practices
**Triage Decision:** Valid — standard best practice gap.

None of `deposit`, `transfer`, or `withdraw` emit events. Off-chain tools
(block explorers, monitoring dashboards, subgraphs) have no way to index
balance changes or detect the underflow attack described in Finding 1 without
full transaction tracing.

Standard events for this pattern:

```solidity
event Deposit(address indexed sender, uint256 amount);
event Transfer(address indexed from, address indexed to, uint256 amount);
event Withdrawal(address indexed sender, uint256 amount);
```

Not a security vulnerability on its own, but a required best practice for
any production contract. Classification as Best Practices is correct.
